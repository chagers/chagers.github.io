Procs, Lambdas, Blocks, and a Cooking Robot

Wait... What?

05.17.15

In today's tech blog I will be talking about what procs, lamdas and Blocks are, and how they are related with a little help from a cooking robot friend.

First, let's look at what each of these are:

  • Procs (short for Procedures) are objects.
  • Lambdas are objects.
  • Blocks are syntax.

Wait, what? I thought everything in Ruby was an object! Nope, not this time. But... by using procs and lambdas we can turn code block into an object. Oh clever Ruby.

Next, let's look at what each of these are made of:

  • Procs take in code block to define what that proc will do when it's called.
    • So.. Say we had a cooking robot, and all we had to do was give it a recipe and it would know how to execute the dish. Defining a proc would be like saving our favorite recipe for chicken vesuvio in the cooking robot, simply calling out "chicken vesuvio" and cooking robot would know what to start cooking! Yum.
  • Lambdas exist in the Proc class, and also take code block to define what the lambda will do when it's called.
    • So.. You're probably thinking, well ok same thing.. Why the different name? We'll look more at that in the following section.
  • Blocks are made up of syntax that tells methods, procs, lambdas, loops, etc. what to do.
    • If we're still thinking along the lines of our cooking robot example, a block would represent the recipe for chicken vesuvio.

Now, what good are they?

  • As we've already seen above, procs and lamdas comprised of code block make excellent shortcuts to running specific tasks - like recipes for the cooking robot.
  • But what is the difference between a proc and a lambda? Well this is where it gets a bit trickier:
    • Say we wanted to give our cooking robot a recipe for cinnamon rolls. In this particular recipe, there is a sub-recipe for the rolls themselves, and another for the frosting. What would happen if we created both of these recipes as procs?
    • Well, if both were procs, and we indicated that we wanted to return the results after each sub-recipe was complete to check the progress, we'd get some unintended results. Using return with a proc would mean that the cooking robot would exit the entire recipe at whatever point return was used. Which may very well mean we'd get rolls without frosting! Well, that's no good. How can we fix this? Well, by using lambdas, of course (I know that's what you already guessed).
    • If we were to write the rolls and icing recipes as lamdas, we could use return on those and the cooking robot would return the recipes at that moment in time before it continued on its merry way to finishing the entire cinnamon roll recipe. This would allow us to check in, but would not stop the robot from completing the (very important) tasks at hand.

In review:

  • Procs and lambdas are saved snippets of code block for later use.
  • Using return on procs means exiting from its code block as well as the method that contains the proc. (Like making cinnamon rolls without frosting.)
  • Using return on lambdas means exiting from its code block but giving control back to the method to continue execution. (Like pausing to taste the frosting, but continuing on to create the finished product.)

So - there you have it. The classic cooking robot example to give a little bit more insight into the differences between procs and lambdas, and how code block relates to (and creates!) them. Now, off to cook up some more code.
